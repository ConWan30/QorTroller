## Executive Summary

The VAPI grind (100 consecutive_clean sessions required for Stage 1 graduation) has a
**content-blindness gap**: the protocol cannot distinguish competitive NCAA CFB 26 gameplay
from menu navigation or idle controller sessions. A session where the player navigates menus
for 5 minutes counts identically toward consecutive_clean as a session of active gameplay.
This gap is empirically confirmed by 6 passing audit tests (see `test_phase235_gad_audit.py`).

**Gap severity:** HIGH for grind integrity. The GIC chain remains cryptographically sound, but
the 100 sessions it chains may include zero seconds of actual competitive gameplay.

## Task 1.1 — Session Boundary Definition

### What is a "session" in ruling_validation_log?

A **session** in the grind context (a row in `ruling_validation_log`) is NOT a time window of
continuous HID capture. It is exactly one processed `ruling_request` event.

**Session lifecycle:**
```
1. OPEN:    ruling_request event written to agent_events table
            Sources:
              (a) BridgeAgent tool request_adjudication      bridge_agent.py:3703-3704
              (b) BridgeAgent tool request_live_adjudication  bridge_agent.py:3842-3843
              (c) POST /agent/adjudicate HTTP endpoint        transports/http.py:851-852
            NOT auto-generated by _session_loop

2. PROCESS: SessionAdjudicator.run_event_consumer() polls every 300s
            session_adjudicator.py:20,140
            Processes via _process_ruling_request(event)   session_adjudicator.py:184

3. EVIDENCE: _process_ruling_request() reads the last 20 PoAC records at adjudication time
             records = self._store.get_recent_records(limit=20, device_id=device_id)
             session_adjudicator.py:194
             Evidence fields extracted:
               hard_cheat_codes: inference in {0x28, 0x29, 0x2A}  session_adjudicator.py:200
               advisory_codes:   inference in {0x2B, 0x30, 0x31, 0x32}  session_adjudicator.py:201
               record_count, enrollment_status, avg_humanity, risk_label, l6b_probes
               session_adjudicator.py:204-213

4. RULING:   LLM (claude-opus-4-6) or _rule_fallback() produces (verdict, confidence, reasoning)
             Stored in agent_rulings table

5. VALIDATE: SessionAdjudicatorValidationAgent processes agent_ruling →
             calls _rule_fallback(evidence) → computes divergence →
             inserts into ruling_validation_log with pcc_state+pcc_host_state
             session_adjudicator_validator.py:144-234

6. CLOSE:    Row in ruling_validation_log is permanent.
             pcc_state=NOMINAL + pcc_host_state=EXCLUSIVE_USB + divergence=0 → counts
```

**Key constraints:**
- A session is manually triggered — the operator or BridgeAgent calls `request_adjudication`
- Evidence = snapshot of the last 20 PoAC records at the moment of adjudication
- There is NO time-boundary concept (a session is not "the last N seconds")
- Multiple sessions per bridge run: yes, each `ruling_request` event is independent
- _session_loop runs continuously at 1.0s intervals (dualshock_integration.py:357) and
  produces PoAC records — but does NOT write ruling_request events

**Evidence blindspot:** The 20 PoAC records in evidence contain inference codes (hard cheats,
advisories) derived from the PITL stack. They do NOT contain raw HID frame data, game state
information, or any indication of whether the player was in a competitive match or a menu.

## Task 1.2 — Content-Blind Session Counting Verification

### Hypothesis
> "A menu-only session will count toward consecutive_clean identically to a gameplay session."

### Methodology

Two HID input profiles were defined and traced through the full adjudicator pipeline:

**STREAM A (gameplay-like):**
- Continuous stick deflections (left stick: varied azimuth, R2: sustained press)
- Frequent L2/R2 trigger activations (football play calls, sprinting)
- Varied face button presses (X/Square/Triangle/Circle)
- Active IMU motion (micro-tremor + grip shifts during play)
- Expected PoAC inference: 0x20 (INFER_NOMINAL) — human gameplay within calibration bounds

**STREAM B (menu-like):**
- D-pad only navigation (no analog stick input → sticks at neutral 128/128)
- Sparse button presses (X to confirm, Circle to back) with multi-second gaps
- No L2/R2 trigger activation
- Minimal IMU (controller at rest or desk-held without motion)
- Expected PoAC inference: 0x20 (INFER_NOMINAL) — no PITL layer has a semantic "menu" signal

### Evidence Pipeline Trace

```
STREAM A → 20 PoAC records (inference=0x20 each)
  → get_recent_records(limit=20) returns 20 rows
  → hard_cheat_codes = []    (0x28/0x29/0x2A: not present)
  → advisory_codes   = []    (0x2B/0x30/0x31/0x32: not triggered by normal gameplay)
  → evidence = {"hard_cheat_codes": [], "advisory_codes": [], ...}
  → _rule_fallback(evidence) = ("FLAG", 0.05, "No anomalies detected")
  → LLM returns FLAG(0.05) (no signals) → llm_verdict = fallback_verdict
  → divergence = (FLAG != FLAG) AND (|0.05 - 0.05| > 0.30) = False
  → pcc_state=NOMINAL, pcc_host_state=EXCLUSIVE_USB → pcc_ok=True
  → consecutive_clean += 1  ✓

STREAM B → 20 PoAC records (inference=0x20 each)
  → get_recent_records(limit=20) returns 20 rows
  → hard_cheat_codes = []    (menu navigation: no injection/wallhack/aimbot)
  → advisory_codes   = []    (no L4/L5 trigger for in-bounds menu input)
  → evidence = {"hard_cheat_codes": [], "advisory_codes": [], ...}
  → _rule_fallback(evidence) = ("FLAG", 0.05, "No anomalies detected")
  → LLM returns FLAG(0.05) (no signals) → llm_verdict = fallback_verdict
  → divergence = False
  → pcc_state=NOMINAL, pcc_host_state=EXCLUSIVE_USB → pcc_ok=True
  → consecutive_clean += 1  ✓
```

### Empirical Confirmation

All 6 audit tests pass (see `bridge/tests/test_phase235_gad_audit.py`):

## Task 1.3 — L4/L5 Feature Sensitivity Analysis

### L4 (0x30 BIOMETRIC_ANOMALY) — Mahalanobis Biometric Fingerprint

L4 was calibrated on N=74 gameplay sessions (hw_005–hw_078) with active controller use.
Thresholds: anomaly=7.009, continuity=5.367 (mean+3σ / mean+2σ on gameplay distribution).

**Feature behavior during menu navigation:**

| Feature | Gameplay range | Menu-like value | L4 sensitivity |
|---------|---------------|-----------------|----------------|
| trigger_onset_velocity_L2 | 0.1–0.8 | ~0.0 (no L2 presses) | Deviates toward 0 |
| trigger_onset_velocity_R2 | 0.1–0.8 | ~0.0 (no R2 presses) | Deviates toward 0 |
| micro_tremor_accel_variance | 200k–400k LSB² | 200k–300k (physiological tremor still present) | STABLE |
| grip_asymmetry | player-specific | player-specific (grip unchanged) | STABLE |
| stick_autocorr_lag1 | variable (active deflection) | ~1.0 (neutral 128/128, no movement) | HIGH DEVIATION |
| stick_autocorr_lag5 | variable | ~1.0 | HIGH DEVIATION |
| tremor_peak_hz | player-specific | player-specific (physiological) | STABLE |
| tremor_band_power | player-specific | player-specific | STABLE |
| accel_magnitude_spectral_entropy | 3.0–8.6 bits | 3.0–5.0 bits (less motion) | Mild deviation |
| press_timing_jitter_variance | 0.001–0.05 | Very low (sparse presses, uniform gaps) | DEVIATION |

**L4 verdict for menu navigation:**

Stick autocorr (lag1/lag5 both ~1.0 vs calibration mean) and zero trigger velocities are the
dominant deviations. The combined Mahalanobis distance MAY exceed 7.009, potentially triggering
0x30 BIOMETRIC_ANOMALY advisory. However:

1. Even if 0x30 fires → advisory_codes=[0x30] → `_rule_fallback` returns FLAG(0.5)
2. LLM would also return FLAG for a biometric advisory → same verdict → divergence=0
3. Session still counts toward consecutive_clean (T235-GAD-AUDIT-5 confirms)

**L4 cannot serve as a gameplay discriminator** because:
- 0x30 means "biometric anomaly vs personal baseline" — not "not playing the game"
- A player with low trigger activity (conservative playstyle) would also trigger 0x30
- L4 is intra-player anomaly detection, not game-state classification

### L5 (0x2B TEMPORAL_BOT) — Temporal Rhythm Oracle

L5 computes button press inter-beat intervals (IBI) and derives: CV, Shannon entropy,
quantization artifact score. Fires 0x2B when patterns resemble ML bot timing.

**Feature behavior during menu navigation:**

- **IBI sequences**: sparse menu presses (one press every 2–5 seconds) produce long, potentially
  uniform IBIs. This superficially resembles an ML bot that fires at regular intervals.
- **CV of IBIs**: low variance (menu player presses at similar rates) → low CV → bot-like
- **Shannon entropy**: fewer unique IBI values (fewer button types) → lower entropy
- **Quantization artifacts**: 0.0 (human doesn't quantize to 16ms grid)

**L5 verdict for menu navigation:**

L5 MAY fire 0x2B for highly regular menu navigation. However:

1. Even if 0x2B fires → advisory_codes=[0x2B] → `_rule_fallback` returns FLAG(0.5)
2. Same verdict logic as L4 case → divergence=0 → still counts (T235-GAD-AUDIT-6 confirms)

**L5 cannot serve as a gameplay discriminator** because:
- 0x2B means "temporal pattern resembles bot" — not "not in competitive match"
- A human navigating menus with regular spacing triggers 0x2B for the wrong reason
- False positive risk: a careful menu player gets the same flag as an ML bot

### Summary: No PITL layer can signal "not competitive gameplay"

| Layer | Code | Triggered by menu? | Verdict if triggered | Counts toward grind? |
|-------|------|-------------------|---------------------|---------------------|
| L2 | 0x28 | No (no IMU injection) | BLOCK | No (hard gate) |
| L3 | 0x29/0x2A | No (no ML bot pattern) | BLOCK | No (hard gate) |
| L4 | 0x30 | Maybe (stick/trigger deviance) | FLAG | YES |
| L5 | 0x2B | Maybe (sparse regular presses) | FLAG | YES |
| L2B | 0x31 | No (disabled by default) | FLAG | YES (advisory) |
| L2C | 0x32 | No (dead zone in NCF CFB 26) | — | N/A |

**Critical finding:** Hard gate codes (0x28/0x29/0x2A) that would prevent grind advancement
require active cheat injection — menu navigation is clean human input that triggers no hard codes.
Advisory codes (0x30/0x2B) may fire but produce FLAG verdicts that don't break the streak.

## Task 1.4 — Protocol-Purpose Position

### Current unit: "HID session where bridge captured input"

The current protocol treats a "session" as any discrete adjudication event backed by 20 PoAC
records from any controller activity while the bridge is running. This is the correct unit for
the anti-cheat purpose (detecting bots during actual play), but it is the WRONG unit for the
grind integrity purpose (certifying 100 legitimate competitive gameplay sessions).

### The gap in protocol purpose

**Anti-cheat purpose** (correct use of consecutive_clean): "Is this human input, not a bot?"
The PITL stack is well-suited to this: PoAC records + inference codes + PCC attestation together
certify human physical presence with a connected controller.

**Grind integrity purpose** (incorrect assumption in consecutive_clean): "Did this human
engage in 100 actual competitive gameplay sessions?"
The PITL stack CANNOT answer this: there is no game-state input to any VAPI layer.

### Why this matters for the grind

**Attack scenario:**
1. Controller connected, bridge running, EXCLUSIVE_USB confirmed
2. Player navigates main menu for 5 minutes (selecting game modes, checking stats)
3. Operator calls `request_adjudication`
4. Evidence: 20 NOMINAL records (no cheats, no advisories) → FLAG(0.05) → divergence=0
5. Session counts. GIC stamped. Chain advances.
6. Repeat 100 times without ever entering a competitive match.
7. GIC_100 hash is cryptographically valid — but over 100 menu sessions.

**What the Phase 235 protocol guarantees (correctly):**
- Human physical presence (PCC NOMINAL + EXCLUSIVE_USB)
- No cheat software active (hard codes)
- Human input patterns (biometric fingerprint within range)
- Cryptographic continuity (GIC chain integrity)
- LLM/fallback agreement (validation divergence)

**What the Phase 235 protocol does NOT guarantee:**
- The game was in competitive match mode
- Any gameplay action occurred (triggering, stick deflection, button combos)
- The session was of competitive-use duration

### Protocol-purpose gap statement

> "The consecutive_clean gate certifies that a human controller user generated 100 PITL-clean
> sessions. It does not certify that those sessions contained competitive gameplay activity.
> The grind produces a cryptographic proof of human presence, not of human gameplay."

This is a meaningful distinction for tournament integrity: a grind completed entirely in menus
produces a GIC chain with identical cryptographic properties to a grind of 100 competitive matches.
