## 1. The VPM Definition and Protocol Purpose

VAPI does not merely output "HTML." VAPI produces **Verified Projection Media
(VPM)**.

A Verified Projection Media artifact is a deterministic, human-operable
projection compiled from canonical VAPI state, proof manifests, and frozen
protocol primitives.

A VPM may:

- explain verified state,
- compress verified state into a human-operable surface,
- visualize proof limitations,
- package verified state for a stakeholder workflow,
- distribute a verifiable view of already-existing protocol truth.

A VPM may not:

- become source of truth,
- strengthen a proof claim,
- hide proof limitations,
- change runtime authority,
- bypass AgentScope / Cedar policy,
- bypass governance,
- convert a demo or draft artifact into production truth.

The core protocol equation for this layer is:

```text
Canonical Cryptographic Truth
-> Deterministic Proof Media
-> Audience-Specific Trust
```

HTML is the first medium through VBDIP-0002. VPM is the broader media category
for future deterministic projections such as QR codes, broadcast overlays,
hardware certificates, PDF dispute packets, wallet cards, and public protocol
boards. VPM inherits the strict epistemic discipline of VAPI evidence design:
it may explain proof state, but it must never alter proof state.

## 3. VPM Lifecycle

Every VPM follows this lifecycle:

```text
Canonical State
-> VPM Wrapper Manifest
-> Deterministic Compile
-> Integrity Label
-> Stakeholder Surface
-> Expiry / Revocation Check
```

Lifecycle semantics:

- **Canonical State**: proof roots, ZKBA manifests, FROZEN-v1 primitive hashes,
  consent state, anchor state, and operator-agent state.
- **VPM Wrapper Manifest**: a VPM-specific wrapper that references existing
  proof manifests and compiler outputs; it does not replace them.
- **Deterministic Compile**: a content-addressed projection generated by an
  approved compiler version.
- **Integrity Label**: mandatory visible claim summary and limitation surface.
- **Stakeholder Surface**: audience-specific media such as a QR pass, player
  wallet card, hardware certificate, or dispute packet.
- **Expiry / Revocation Check**: final validity check before display or use.

The existing GIC Continuity Ledger is a ZKBA compiler target. It is not yet an
active VPM artifact until a VPM wrapper manifest, Integrity Label, and VPM
visual grammar tests exist for it.

## 5. Visual Honesty Grammar (VPM-HONESTY-001)

All VPM artifacts must obey a protocol-native visual grammar that makes
overclaiming visually difficult.

Visual states:

- `live` = saturated colors
- `dry-run` = striped patterns
- `emulated` = desaturated / greyscale
- `frozen-disabled` = locked iconography
- `revoked` = crossed out / redacted
- `unverified` = high-contrast warning bands

This section intentionally does **not** use a `VED-INV-N` identifier. In the
current VEDIP-0001 mapping, the tenth VED alias already refers to `INV-010` /
`L6B_ENABLED default=False`. VPM visual honesty therefore uses a local draft
identifier, `VPM-HONESTY-001`, until or unless a future PV-CI invariant is
explicitly added through governance.

Every VPM must include an Integrity Nutrition Label detailing:

- Proof Type,
- Capture Mode,
- Raw Biometrics Exposed (Yes / No),
- Consent Active,
- ZK Verified,
- On-Chain Anchor status,
- Proof Weight,
- Revocation Status,
- Limitations.

## 6. Stakeholder Utility Layer

One canonical truth may generate multiple audience-specific projections.

| Stakeholder | Claim shown | Private data not exposed | Verifier action | Failure state |
|---|---|---|---|---|
| Gamers | VHP credentials, practice streaks, proof wallet status, hardware certificates | Raw 13-dimensional biometric telemetry, session traces, centroids, covariance | Inspect Integrity Label and local proof status | Wallet card renders warning / revoked / expired |
| Developers | Trust SDK pages, verified-human matchmaking sandbox examples | Raw HID streams and biometric vectors | Verify wrapper manifest hash and ZKBA manifest hash | SDK page shows integration unavailable |
| Manufacturers | Hardware lineage certificates, module epochs, firmware provenance, proof-weight tier contribution | Player biometrics and private gameplay history | Verify hardware cert ID, firmware hash, and proof-weight limitation | Certificate renders downgraded or unverified |
| Tournament Organizers | QR eligibility pass, check-in status, dispute packet availability | Raw biometrics, private practice habits, non-consented telemetry | Scan QR, verify manifest, check anchor/expiry/consent | Pass renders ineligible, stale, revoked, or dispute-required |
| Marketplace Buyers | ZKBA market card, consent status, tier limitations, revocation state | Raw proof inputs and non-consented data | Verify listing wrapper, Curator review state, and proof-weight | Listing renders quarantined or buyer-warning state |
| Governance / Deployer | Operator accountability card and draft/action review surface | Non-relevant operator internals and raw biometric payloads | Verify Cedar lane, decision provenance, and signature lineage | Card renders lane violation or authority unavailable |

## 7. Failure-State Rules

VPM artifacts fail visibly. They do not fail silently.

Required failure states:

- **Missing manifest**: render `unverified`; no eligibility, listing, or wallet
  success state may appear.
- **Compiler hash mismatch**: render `verification-unavailable`; do not show
  `verified`.
- **Proof-weight omission**: fail compilation; no VPM artifact is emitted.
- **Revoked consent**: render `revoked`; marketplace and data-buyer surfaces
  become invalid.
- **Stale verification key**: render `warning`; verifier must refresh or reject
  according to the consuming workflow.
- **Absent anchor**: render `not anchored`; do not imply on-chain finality.
- **DEMO**: render visible demo watermark or equivalent visual state.
- **FROZEN_DISABLED**: render locked / disabled; never render as active.

## 8. Curator Marketplace Lane

Curator is the Operator Agent responsible for VPM marketplace and artifact
coherence. Curator monitors:

- ZKBA manifest completeness,
- VPM wrapper manifest completeness,
- proof-weight classification,
- consent validity,
- compiler hash matching,
- buyer-facing visual honesty,
- revocation and quarantine state.

**In O1_SHADOW:** Curator may observe and draft review VPM artifacts only.

**Future authority:** Any listing mutation, tier update, quarantine action, or
IPFS pinning of a VPM requires explicit Cedar / AgentScope expansion through
governance.

## 9. AI Role Constraint

AI agents do not create protocol truth.

Under VBDIP-0002A, AI agents may:

- detect inconsistency,
- compile deterministic VPMs through approved tooling,
- flag visual misrepresentation,
- summarize verification outcomes,
- draft operator review notes.

AI agents may not:

- fabricate proof claims,
- bypass Cedar policy,
- alter proof weight,
- convert demo artifacts into production VPMs,
- change source-of-truth state,
- hide failure states.

## 10. VPM Projection Registry

This sidecar reserves VPM identifiers to prevent derived-artifact sprawl. All
future artifacts must map to one registered identifier and one lifecycle state.

Lifecycle ladder:

```text
Reserved -> Draft Manifest -> Compiler Target -> Test Fixture -> Active
```

Lifecycle rules:

- `Reserved` means the identifier exists only to prevent naming drift.
- `Draft Manifest` means a wrapper manifest shape exists but no compiler target
  is active.
- `Compiler Target` means deterministic compilation exists for that VPM ID.
- `Test Fixture` means representative fixtures and visual honesty tests exist.
- `Active` means governance has approved use in the relevant workflow.

All stakeholder-facing VPM IDs remain `Reserved` until a later implementation
commit supplies manifest support, compiler support, and tests.

## 12. Decision Blocks

**Decision L1 - VPMs as Human-Operable Surfaces**

*Resolved:* VPMs are deterministic human-operable surfaces for cryptographic
state, not protocol source of truth.

**Decision L2 - Anti-Hype Visual Grammar**

*Resolved:* Visual honesty is protocol law. VPMs must use mandatory visual
states to reflect technical limitations.

**Decision L3 - VPM Registry Discipline**

*Resolved:* All projections must use registered VPM identifiers to prevent
un-auditable artifact sprawl.

**Decision L4 - AI Stewardship Boundary**

*Resolved:* AI agents compile VPMs to explain verified state, but may never
create or escalate protocol truth.

**Decision L5 - VPM Wrapper Discipline**

*Resolved:* VPM wraps ZKBA and proof manifests; it does not replace frozen
schemas, compiler constants, or primitive commitments.

**Decision L6 - Registry Lifecycle Discipline**

*Resolved:* Reserved VPM IDs are not shipped features. A VPM ID becomes active
only after wrapper manifest, compiler target, test fixture, and governance
approval exist.

**Decision L7 - Failure-State Visibility**

*Resolved:* Invalid, incomplete, stale, revoked, demo, or disabled proof state
must render visibly degraded.
