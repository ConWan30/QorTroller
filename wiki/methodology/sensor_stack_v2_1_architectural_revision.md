## Executive Summary

The v2.0 sensor-stack ideation proposed seven DualSense Edge surfaces as candidates for L4 feature inclusion. A methodology-research verification pass (Track 1 prerequisite research) applied the BT-CALIB-LESSON-001 verification-block discipline to each surface — vendor-spec confirmation from the DualSense reverse-engineering corpus, open-source driver validation against `hid-playstation.c`, independent literature anchor — and produced a per-surface decision matrix with one position promoted to PRIMARY DISCRIMINATOR, one position falsified at default scope on privacy grounds, three positions held at CO-SIGNAL tier, and two positions held at ADVISORY tier with explicit constraints.

This v2.1 revision preserves the broader Track 1 framing — VAPI is structurally first in the DualSense Edge sensor-set surface coverage against the ARES 2024 audit cohort (BattlEye / EAC / FACEIT / Vanguard) and the RICOCHET Season 02 + Season 05 host-OS-side rootkit-class instrumentation — and locks in the six surface-tier assignments the research established, with Stage A pre-corpus measurement gates explicit before the v2 L4 architecture spec exits draft state.

The single most consequential positive finding: adaptive trigger force-curve telemetry is the leading L4 v2 candidate, with literature-anchored inter-subject EER bracketing the Cross-button NCAA fingerprint (1% EER lab-best per Saevanee 2009 piezo-NN; 5-15% EER realistic field expectation; 76.8-89.6% identification on driving pedal force corpora per Wakita 2006 and Miyajima 2007), and adversarial spoof cost materially higher than any other surface in the matrix because translator-class hardware (Cronus Zen, XIM Matrix, reWASD) does not synthesize biomechanically-structured continuous force curves.

The single most consequential negative finding: microphone array is dropped from default scope on privacy-falsification grounds, with concrete BIPA / GDPR / CIPA litigation anchors and per-scan statutory damages establishing that any biometric capture surface with non-consenting-third-party exposure (household members, roommates at tournament-from-home) is structurally falsified regardless of engineering merit. The Cruz v. Fireflies AI analog is the controlling case: non-account-holding meeting participants whose voice was captured without consent maps directly onto household third parties at a tournament-from-home participant.

The recommendation from the canonical anchor stands: lock Surface 1 as v2 L4 PRIMARY discriminator now (Stage A engineering allocation); run Empirical Unknowns #1 and #4 measurement campaigns before v2 L4 architecture spec is drafted (Stage A measurement gate); implement adaptive-trigger challenge-response as L4-coupled-to-L6 in Stage B (after Stage A measurements land).

## Section 2 — Surface 1 (Adaptive Trigger Force-Curve): PRIMARY DISCRIMINATOR Promotion

The DualSense L2/R2 triggers are mechatronic actuators with internal voice-coil motors (Sony codename "Walther" per PSDevWiki) capable of programmable resistance plus continuous analog position/force telemetry. The HID input report encodes trigger axis at byte 5 (L2) and byte 6 (R2), 8-bit unsigned, neutral 0x00 → fully depressed 0xFF, reported every input report. On the DualSense Edge over USB the default polling rate is 1000 Hz (PCGamingWiki Controller:DualSense Edge entry verified by user Yuuyatails 2024-08-27; DSX developer statement on Steam Community), versus ~250 Hz on standard DualSense over USB and 133-250 Hz over BT depending on host stack and firmware.

Critical capability for challenge-response: the host can read back the user's applied force *while a trigger effect is active*. DualSense-Windows (Ohjurot) explicitly documents "Controlling the adaptive triggers (3 Types of effects) and reading back the users force while active." pydualsense and dualsense-controller-python expose the same surface. HID output report fields (USB report ID 0x02, 47-byte report; BT report ID 0x31, 78-byte report) carry trigger effect commands per Game Controller Collective Wiki `FFB_Trigger_Effect_Factories`: `Off`, `Feedback`, `Weapon`, `Vibration`, `Slope`, `MultiplePositionFeedback`. Each effect parameterizes resistance start position (0-9 in trigger travel zones), end position, strength (1-8), and frequency for vibration mode. This is a programmable-resistance challenge channel.

Open-source driver validation: `drivers/hid/hid-playstation.c` (Colenbrander, Linux 5.12+) parses both USB report 0x01 and BT report 0x31. The kernel struct `dualsense_input_report_common` carries `rx` (L2) and `ry` (R2) byte fields. Output report struct `dualsense_output_report_common` carries `right_trigger_motor_mode` and `left_trigger_motor_mode` plus 10-byte parameter blocks per trigger.

Per-firmware-version differences: firmware 0630 (2025-09-17) added multi-device BT pairing without changing trigger input encoding (PCGamingWiki). The Edge revision (CFI-ZCP1, USB PID 0x0DF2) preserves the 8-bit trigger encoding but ships with Hall-effect trigger sensors rather than the original DualSense's potentiometer-based trigger position sensing (gamepadtest.app PS5 controller tester documentation). **This means trigger position telemetry on the Edge is drift-resistant and its reported value is closer to true mechanical position than on a worn standard DualSense.** This is a positive consequence for the v2 L4 feature reliability.

The literature anchor layers across three biometric domains, all measuring inter-subject separability on continuous-force tasks:

| Domain | Source | Result |
|--------|--------|--------|
| Driving-simulator pedal force | Wakita 2006 (IEICE) | 81% ID on N=12 simulator drivers, 73% on N=30 real-vehicle |
| Driving-simulator pedal force | Miyajima 2007 (Proc. IEEE) | 76.8% ID on N=276 drivers (real city), 89.6% in simulator |
| CAN-bus pedal/steering | Enev 2016 (PoPETs) | ~100% ID on N=15 from brake-pedal alone (replication caveat: Gahr 2018 did not reproduce on naturalistic data) |
| CAN-bus uncontrolled | Fugiglando 2019 (IEEE TITS) | 76% 2-driver, ~50% 5-driver on N=64 uncontrolled |
| MIDI key-velocity | Van Vugt 2013 (Frontiers Psy) | 100% recognition N=8 within-session; separation ratio ~2.3 (inter-class 7ms vs intra-class 3ms) |
| MIDI key-velocity | Stamatatos 2005 (AI 165) | ~70% ID on N=22 pianists, inter-piece, vs ~5% chance |
| Keystroke pressure | Saevanee 2009 (CCNC) | 1% EER on N=10 single-session (treat as upper bound) |
| Keystroke pressure | Antal 2015 (Procedia Tech) | 12.9% EER on N=42 multi-session (down from 15.3% timing-only) |
| Keystroke pressure | Chang 2012 (JSS) | 12.2% timing-only → 6.9% with pressure on N=100 + 20 imposters |
| Keystroke pressure | Piezo-NN | 1.09% EER with neural-network fusion of force + timing |

Realistic field expectation for a held-trigger-pull-only feature set in the same-platform, multi-player setting: EER 5-15% per pull, with separability strongly dependent on whether the pull is during a high-information event (firing weapon, accelerating) or a passive low-amplitude event. This brackets the Cross-button transition-timing fingerprint (NCAA Football 26 corpus) and exceeds it in lab-best conditions (1% EER), degrades in the field (Fugiglando 2019 5-driver uncontrolled accuracy ~50%). On the protocol's separation-ratio framing, force-curve pull events clear the 0.5 floor decisively.

Separation-relevant features (literature-supported): peak force amplitude, ramp slope (dF/dt during onset), plateau hold time at fully-depressed position, release velocity, force-time integral (F·dt), pull duration, count and timing of micro-modulations during sustained pull (related to physiological tremor at 8-12 Hz per Lakie et al. 2012).

Adversarial spoof cost: very high. Cronus Zen, XIM Matrix, and reWASD synthesize button/stick/trigger values via macro/script engines but do not synthesize force-vs-time curves with realistic biomechanical structure. Cronus Zen "Expert Controller GamePack" exposes Right/Left Trigger Deadzone Min/Max and Custom Analog Stick Curve with 10 configurable points — parametric output remapping, NOT biomechanical trace synthesis. RICOCHET Anti-Cheat Season 02 (callofduty.com/blog/2026/02/call-of-duty-black-ops-7-ricochet-anti-cheat-season-02, launched February 5, 2026) explicitly targets "machine-modified inputs" by analyzing input *behavior* (timing, consistency, response patterns) — confirming the threat model and confirming that no public translator-class device today produces convincing force-curve fakes. Producing realistic continuous force replay would require either physical actuation of the trigger by a robotic finger driving a real DualSense, or a deeper firmware-level intercept that synthesizes byte-level trigger telemetry mid-stream — neither publicly demonstrated against any continuous-force biometric system in the surveyed literature.

**Recommended L4 inclusion tier: PRIMARY DISCRIMINATOR.** Stage A measurement gate (Empirical Unknown #1) required before v2 L4 architecture spec exits draft state: confirm intra-player vs inter-player Mahalanobis separation ratio > 1.0 on N=10 players × 100 trigger pulls × 3 game contexts. If separation ratio > 1.0, force-curve graduates to primary discriminator status; if 0.5-1.0, demote to co-signal; if < 0.5, drop and revisit fusion strategy.

**Adaptive trigger as L4-coupled-to-L6 challenge-response (Stage B):** host issues a `Slope` or `Vibration` resistance profile, observes the player's force-vs-time response over the next 200-500 ms, validates against an enrollment model. This converts the trigger surface from a passive feature source into an active anti-cheat challenge channel. Structurally identical to the L6 haptic challenge channel currently shipping in the protocol, with the additional advantage that the response signal (player force) is bounded by physiological force-velocity envelopes that are not trivially synthesizable by current translator-class hardware.

## Section 3 — Surfaces 2, 4, 5, 6 (CO-SIGNAL and ADVISORY Tiers)

**Surface 2 — Touchpad (CO-SIGNAL):** The DualSense touchpad is a 2-point capacitive surface. Per the parsed HID descriptor (nondebug/dualsense), each finger reports `Index : 7-bit, NotTouching : 1-bit, FingerX : 12-bit, FingerY : 12-bit`, two fingers per touch packet plus an 8-bit Timestamp. **No native pressure proxy or contact-area field is exposed.** This is position-only encoding with binary contact state. Frank et al.'s Touchalytics features include "area covered by fingertip" and "fingertip pressure" — the two highest-mutual-information features in Frank et al. — neither of which is reported by the DualSense touchpad's HID stream. The 12-bit position resolution at the input-report rate is sufficient for stroke-velocity, multi-touch coordination, and inter-touch timing features. Losing the two pressure features shifts realistic EER from Frank's 2-3% range toward 5-10%. Separation ratio remains comfortably above the 0.5 floor. **The binding constraint is data volume per session, not per-event separability.** The DualSense touchpad is rarely engaged during competitive gameplay (primarily menu/map navigation surface), so per-session feature volume is small. This positions touchpad as a co-signal that contributes when engaged but cannot be relied on as a primary discriminator. Adversarial spoof cost: moderate (translator-class hardware typically remaps or ignores touchpad; synthesizing realistic two-finger swipe geometry over time is feasible at the byte level because the surface is position-only with no physiological constraint encoded).

**Surface 4 — Lightbar (CO-SIGNAL, challenge-response witness only):** The DualSense lightbar is a 24-bit RGB LED ring around the touchpad, host-controllable via output report. Linux mainline `hid-playstation.c` registers it as a multicolor LED via `led_classdev_mc` with sub-LEDs for `LED_COLOR_ID_RED/GREEN/BLUE` (Colenbrander patch series, "[PATCH v6 1/4] HID: playstation: add DualSense lightbar support"). Brightness levels exposed: Bright, Mid, Dim (3 levels per the `LightBrightness` enum in Game Controller Collective Wiki Data Structures). Update rate is bounded by the host output-report rate (~250 Hz over USB at standard polling, ~1000 Hz over the Edge's 1 kHz USB polling). Lightbar PWM frequency for the LED hardware itself is not externally specified by Sony, but downstream RGB-LED PWM controllers in this device class typically run in the kHz range and would be invisible to a 30/60 fps camera.

**Engineering path for third-party witness:** Tournament station camera (referee's phone, e.g., 30 fps 1080p, or a webcam) observes a host-issued lightbar challenge pattern signed with the controller's secret. Practical pattern: 3-color symbol stream at 5-15 symbols/second over a 5-second authentication window, producing 25-75 bits — sufficient for a Merkle-proof challenge response keyed off the controller's session secret. Literature anchor: visible-light covert channels (Hao 2012 COBRA MobiSys, Hu 2013 LightSync MobiCom, Li 2014 HiLight VLCS, Kuo 2014 Luxapose MobiCom, Ghiasi 2021 Passive Light Comms MobiCom) show 30-60 fps cameras can reliably decode visible-light symbol streams from screen-class sources at multi-meter ranges; PWM-modulated RGB LED sources at lower modulation rates are even more permissive.

This is comparable in role to the LAN-tower BlueZ witness path established in the BT calibration prerequisite (`bt_calibration_v1_1_architectural_revision.md` Section 4), with the advantage that the optical witness is **fully passive (camera)** rather than requiring an active BlueZ host on the same RF channel. The lightbar is host-controlled output, not player-discriminative input — it carries zero per-player discriminative information by itself. Its value is as a challenge-response witness channel, not as a feature source. Spoofing the lightbar value at byte level is trivial (translator-class output remap); spoofing the physical optical emission observed by an independent camera that does not connect to the host PC is structurally equivalent to defeating the BT witness path — the cheat would need to either compromise the witness device, add a second emitter at the same spatial location, or maintain real-time MITM between the host and a real controller. Spoof cost vs independent witness: high.

**Surface 5 — Battery drain (ADVISORY ONLY):** Battery telemetry is exposed via the input report's status byte as a 4-bit power level (11 distinct levels, ~10% per step, range 0x00-0x0A per neolit123/go-dualsense-battery analysis). A `PowerState` enum (Discharging, Charging, Complete, AbnormalVoltage, AbnormalTemperature, ChargingError) per Game Controller Collective Wiki. **No exposed instantaneous voltage, current, or drain-rate field over HID** — telemetry is the bucketed level only. The PMIC is a Dialog DA9087 (per psdevwiki); raw instantaneous power is internal to the PMIC and not surfaced. Update cadence: value changes only when level crosses a ~10% bucket boundary — observed real-world cadence is on the order of multiple minutes per bucket transition under normal load. The power side-channel literature (Yan 2017, WISERS 2023, Zhang 2015) requires power-trace sampling rates of kHz to MHz, fed by purpose-built measurement hardware or by polling instantaneous voltage/current battery files exposed by the OS — **the DualSense exposes neither.** At the available telemetry resolution, the signature of "controller held and actively played" vs "controller resting on desk emitting BT keep-alives" is dominated by absolute drain rate over multi-minute windows; observable at the 11-bucket resolution within a 30-60 minute session, but no per-event temporal structure. Useful as a session-integrity check (controller has not been swapped for an idle one) over 30+ minute windows — advisory-only capability. Adversarial spoof cost: trivial (battery state in the input report is unauthenticated; can be set to any plausible value by translator-class firmware).

**Surface 6 — Stick analog noise floor + Hall-effect stick fingerprinting:** Stick X/Y is 8-bit per axis (USB report 0x01 bytes 1-4, neutral = 0x80). The hardware ADC inside the controller is higher-resolution (the SIE CXD9006GG MCU/DSP has multi-channel ADC capability), but the HID encoding truncates to 8 bits per axis. The 8-bit-vs-16-bit debate is settled: at the host HID surface, it is 8 bits, full stop.

**CRITICAL FACT CORRECTION FROM PRIOR IDEATION:** The DualSense Edge ships from Sony with ALPS Alpine **potentiometer-based** stick modules — NOT Hall-effect. The Edge's **trigger** sensors are Hall-effect from factory (gamepadtest.app DualSense tester), but the stick sensors are not. Aftermarket Hall-effect replacement modules (GuliKit, ZeroStick Pro) and TMR modules (MODDEDZONE, XP Controllers, Battle Beaver Customs) exist and are increasingly common in the competitive scene per the product pages.

**Stick noise floor analysis (held vs unheld classifier):** Stick travel maps to ~10 mm of physical thumb displacement at the stick top. At 8-bit resolution that is ~0.04 mm per LSB. Physiological hand tremor at the thumb-on-stick-top pose has displacement amplitudes of order 0.05-0.5 mm peak-to-peak in healthy adults (Lakie 2012 J. Physiology; Allum tremor PubMed 943474). The 8-bit HID encoding is **at the noise floor** — tremor is detectable when the player is gripping with intent (high-amplitude end), but smaller-amplitude tremor signatures will be quantization-limited. A 16-bit encoding would lift tremor signatures clear of quantization but is not exposed. A realistic "controller in hand" vs "controller on desk" classifier using only stick X/Y at 8-bit can detect held vs unheld at very high accuracy (held controllers show non-zero high-frequency variance in the 8-12 Hz band; placed controllers show none). **As a binary held/unheld discriminator, the stick noise floor is reliable. As a per-player fingerprinting feature in an N≈50 same-population tournament, the 8-bit floor is too coarse to support strong claims.** Recommended tier: CO-SIGNAL (held/unheld binary plus tremor-band variance as supporting evidence).

**Hall-effect stick fingerprinting (Givehchian-style physical-layer analog):** Givehchian et al. IEEE S&P 2022 reported 40-47% of mobile devices in field tests had unique BLE physical-layer fingerprints from manufacturing imperfections. The structural analog for Hall-effect stick modules: per-unit response curve (X-vs-magnetic-field, Y-vs-magnetic-field, deadzone shape, return-to-center bias) carries manufacturing-tolerance variation that could constitute a hardware fingerprint. **Same-controller-population separability constraint (this is the L8 BT lesson applied to Surface 6 — it BITES HARD here too):** N≈50 identical DualSense Edge units in a tournament hall, particularly if all running an aftermarket TMR module batch from the same supplier (very common in the competitive Edge scene per the MODDEDZONE, Battle Beaver, and XP Controllers product pages), will sit closer in Hall-stick response-curve feature space than a heterogeneous public population. The Givehchian-style attack achieves 40-47% unique identification on a heterogeneous public population; in a same-model same-batch population the hardware variance budget collapses by an order of magnitude or more. **This surface is therefore marked at the matrix level as session-bound presence attestation only — not cross-session controller identity — until a same-model separability study is run.** Any architectural proposal in v2 that relies on Hall-stick fingerprinting as a long-term controller-identity anchor is superseded by this finding. Recommended tier: ADVISORY ONLY, marked session-bound presence only, blocked from cross-session use until same-model separability study completes.

Adversarial spoof cost on stick X/Y at 8 bits: stick X/Y is the single most heavily-spoofed surface in the entire console-cheat ecosystem (Cronus Zen "Custom Analog Stick Curve" with 10 configurable points, XIM, reWASD). Synthesizing a plausible 8-12 Hz tremor envelope on top of a synthetic stick value is straightforward for any party that has read the physiological-tremor literature; this lowers the bar significantly compared to force-curve synthesis on Surface 1. Spoof cost: low to moderate. This pushes stick noise floor toward co-signal not primary status.

## Section 4 — Surface 3 (Microphone Array): DROPPED at Default Scope (Privacy Falsification)

Two findings compose to produce the DROPPED decision:

**Vendor-spec finding (single-channel reality):** The DualSense exposes audio over USB Audio Class 1.0 (UAC1). Per Cristian Ciocaltea's May 2025 patch series ("HID: playstation: Add support for audio jack handling on DualSense," LWN.net 1022523): "the controller complies with v1.0 of the USB Audio Class spec (UAC1)." On Linux this surfaces as an ALSA card `alsa_card.usb-Sony_Interactive_Entertainment_DualSense_Wireless_Controller-00` with input and output PCM endpoints. Mic only available over USB; built-in microphone is unsupported over BT (PCGamingWiki, BoilingSteam first-look review). The Game Controller Collective Wiki entry describes the controller's microphone path as **"Mono Body Mic with Back Mic sound canceling"** — only a single mono channel is exposed to the host over UAC1, despite the on-controller multi-mic noise-cancelling DSP (Realtek ALC5524 audio codec, Sony codename "Venom," per psdevwiki). **This is a critical finding: the host receives a single mono audio stream, not a multi-mic array.** Most of the room-impulse-response and beam-forming literature requires multi-mic input. The DualSense's on-controller back-mic sound-cancelling DSP destroys the per-mic phase information before the audio reaches the host.

**Privacy falsification (Illinois BIPA + GDPR + CIPA):** Voiceprints are explicitly enumerated biometric identifiers under Illinois BIPA (740 ILCS 14). Concrete 2024-2025 litigation anchors:

- **Beil v. Petco Animal Supplies Stores, Inc.**, Case No. 1:22-cv-06455 (N.D. Ill.) — Judge Sunil Harjani granted preliminary approval March 14, 2024, to a **$445,000 settlement covering 445 warehouse workers** who used a Honeywell Vocollect voice order-picking system. That is $1,000 per class member, the BIPA negligent-violation floor.
- **Parker et al. v. Verizon Communications Inc.**, Case No. 1:24-cv-08436 (N.D. Ill.) — targets Verizon's Voice ID program; Judge Jorge Alonso ordered arbitration May 30, 2025.
- **Cruz v. Fireflies.AI Corp.**, No. 3:25-cv-03399 (C.D. Ill.) — targets Fireflies' "Speaker Recognition" feature which generates voiceprints of *non-account-holding* meeting participants. **The non-account-holding-third-party theory in Cruz is directly analogous to a household member whose audio is incidentally captured by a DualSense mic in a tournament-from-home configuration.**

Statutory damages: $1,000 per negligent violation, $5,000 per intentional/reckless violation, per scan. Even framing the use as "presence attestation, not voice ID" does not avoid BIPA: the trigger is the capture and storage of biometrically-identifiable audio.

EU GDPR Article 9 prohibits processing biometric data for the purpose of uniquely identifying a natural person without one of the narrow lawful bases. The EU AI Act's biometric-categorization rules further restrict any inference of human characteristics from biometric input. US Federal Wiretap Act and state two-party consent regimes (CIPA in California, Florida, Pennsylvania, Illinois 720 ILCS 5/14-2): capture of "oral communications" requires consent; tournament terms-of-service consent does not necessarily reach household third parties whose audio is incidentally captured.

**A narrow non-speech feature subset may survive** (running averages of spectral centroid below the speech band, room reverberation time T60 estimates from impulse responses to controller-emitted speaker challenges, multi-second-binned envelope statistics that do not contain phonetic content and do not reconstruct identifiable voiceprints). However, the processing must be on-device or in a tightly-bounded enclave to avoid raw audio touching infrastructure where consent regimes attach. **The DualSense controller does not provide such an enclave; processing would have to be on the host PC, which means raw audio crosses host-OS boundaries.** There is no published anti-cheat work using even the narrow non-speech feature subset — this would be genuinely novel but is also a privacy-litigation magnet.

**Recommended L4 inclusion tier: DROPPED at default scope; RESEARCH-ONLY for the narrow non-speech feature subset.** The output for this surface is a feasibility-and-privacy-constraints document, not a feature specification. **Do not implement microphone capture in v2 L4.** If the protocol later requires a passive-presence channel, the camera-based optical-witness path on Surface 4 is strictly preferable on every privacy axis (the camera observes only the controller's host-emitted lightbar pattern, not the player or surrounding humans).

## Section 5 — Empirical Unknowns (Pre-Corpus Measurements Required Before v2 L4 Architecture Spec Proceeds)

Eight specific empirical unknowns, each with a decision threshold, must be resolved by pre-corpus measurement before the v2 L4 architecture spec exits draft state.

1. **Adaptive trigger force-curve sample-to-sample autocorrelation in same-player same-game-event windows.** The pull biometric literature is on driving simulators (different actuator), piano keys (different actuator), and smartphone touchscreens (different muscle group). Required: N=10 players × 100 trigger pulls × 3 game contexts (semi-auto fire, full-auto fire, accelerator hold). Measure pull-to-pull intra-player Mahalanobis distance and player-to-player inter-player Mahalanobis distance. **Decision threshold:** if separation ratio > 1.0, force-curve graduates to primary discriminator status; if 0.5-1.0, demote to co-signal; if < 0.5, drop and revisit Surface 6 fusion strategy.
2. **DualSense Edge polling rate stability under host-side load.** The default 1 kHz USB rate is documented but not characterized for jitter under heavy CPU load. Required: jitter histogram at 99th-percentile during a typical Black Ops 7-class game session.
3. **Per-firmware-version trigger telemetry encoding stability.** Firmware 0630 added multi-device pairing; check whether earlier 0500-class and later 0630+ firmwares produce bit-identical trigger byte streams under identical mechanical input.
4. **Hall-effect stick same-model same-batch separability.** Required: N=20 stock Edge units (CFI-ZCP1) plus N=20 of one popular aftermarket TMR module batch (e.g., GuliKit), measure deadzone, return-to-center bias, response-curve linearity, X-Y crosstalk per unit. **Decision threshold:** if unit-level discrimination > 50% rank-1 on heterogeneous models AND > 20% on same-model same-batch, Hall fingerprint moves to co-signal status; otherwise it remains advisory-only and session-bound.
5. **Lightbar PWM modulation frequency.** Sony does not publish; 30/60 fps cameras must capture stable colors, but whether a faster strobe carrier would interfere with the optical-witness symbol stream is unknown.
6. **Touchpad data volume per typical competitive session.** Without a measurement of how many touchpad events occur in a typical 20-minute Warzone or Apex match, the touchpad-as-co-signal contribution to L4 model is speculative.
7. **Battery-state field cadence under continuous heavy haptic + adaptive-trigger + lightbar load.** Test whether the 11-bucket coarse field changes faster than multi-minute under maximum-load conditions; if so, it might rise from advisory to weak co-signal status.
8. **Whether the ALSA mic-stream sample rate floor (UAC1 typically supports 8/16/32/48 kHz) can be set to a sub-speech-band rate by host-side configuration without provoking the controller firmware to renegotiate or fail.** This determines whether the narrow non-speech RIR feature path on Surface 3 is even technically reachable from a Linux/Windows host.

## Section 6 — Same-Controller-Population Separability Constraint (Cross-Surface Matrix)

Per the L8 BT lesson (BT-CALIB-LESSON-001 application context), surfaces whose discriminative variance is dominated by inter-unit manufacturing tolerance collapse in a same-model same-batch population. Applying the constraint to each surface:

| Surface | Variance source | Constraint bites? |
|---------|-----------------|-------------------|
| 1 (trigger force-curve) | player biomechanics, not unit hardware | NO ✓ |
| 2 (touchpad) | player swipe/tap behavior | NO ✓ |
| 3 (microphone) | dropped on privacy grounds; constraint moot | — |
| 4 (lightbar) | challenge-response, no discriminative use | NO ✓ |
| 5 (battery) | advisory only; insufficient resolution for hardware-fingerprint claims regardless of population | NO ✓ |
| 6 (stick Hall fingerprint) | manufacturing tolerance | **YES — surface marked session-bound presence only, blocked from cross-session controller identity until same-model separability study runs** |

The constraint is now formally cross-surface and applies as a global application rule (`CROSS-LESSON-001` in `lessons.md`). Any future architectural proposal naming a controller-internal physical fingerprint feature must explicitly distinguish session-bound presence attestation from cross-session controller identity, and the latter requires a same-model separability study for N≥20 stock units + N≥20 batched-aftermarket units in tournament-realistic conditions.

## Section 7 — Staged Recommendations (Decision-Ready)

**Stage A (immediate, before v2 L4 architecture spec is drafted):**

1. Lock in Surface 1 (adaptive trigger force-curve) as the v2 L4 primary discriminator. Allocate engineering time to a 1 kHz USB capture pipeline that records `Rx` and `Ry` bytes at the polling-rate floor.
2. Run Empirical Unknown #1 (intra-player vs inter-player Mahalanobis distance on trigger pulls) on N=10 players. **Decision threshold:** if separation ratio > 1.0, force-curve graduates to primary discriminator; if 0.5-1.0, demote to co-signal; if < 0.5, drop and revisit Surface 6 fusion strategy.
3. Run Empirical Unknown #4 (Hall-effect stick same-model separability) on N=20 stock + N=20 batched-aftermarket Edge units. **Decision threshold:** if unit-level discrimination > 50% rank-1 on heterogeneous models AND > 20% on same-model same-batch, Hall fingerprint moves to co-signal; otherwise it remains advisory-only and session-bound.

**Stage B (after Stage A measurements land):**

4. Implement adaptive-trigger challenge-response (Surface 1 ↔ L6 coupling): host issues a `Slope` or `Vibration` resistance profile, observes the player's force-vs-time response over the next 200-500 ms, validates against an enrollment model. Converts the trigger surface from a passive feature source into an active anti-cheat challenge channel.

## Section 8 — Constraint Envelope (v2 L4)

v2 L4 includes Surface 1 (adaptive trigger force-curve) as PRIMARY DISCRIMINATOR, gated on Stage A Empirical Unknown #1 separation-ratio measurement. v2 L4 includes Surfaces 2 (touchpad), 4 (lightbar challenge-response), and 6-noise-floor (stick held/unheld) as CO-SIGNALS. v2 L4 includes Surfaces 5 (battery) and 6-Hall-fingerprint (session-bound only) as ADVISORY. v2 L4 does NOT include Surface 3 (microphone array) — dropped on privacy-falsification grounds; narrow non-speech research-only path is documented but not implemented. v2 L4 does NOT claim cross-session controller identity from any surface; all cross-session identity claims are deferred until the relevant same-model separability study completes. v2 L4 detection-floor framing inherits from the broader sensor-stack matrix; novelty-claim framing inherits from the prior-art comparison (none of the audited anti-cheat systems collect any of these six surface types, so VAPI is structurally first in this surface set).
