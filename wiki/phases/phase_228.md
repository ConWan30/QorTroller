## Summary

Phase 228 added biometric presence gating to the `invariant_change` governance category.
When `vhp_gated_invariant_change_enabled=True`, the POST /agent/allowlist-governance-event
endpoint requires a valid `vhp_token_id` in the request body for any invariant_change event.
Missing or expired tokens return 403. The gate fails-open on chain unreachability (prevents
operational outage from testnet connectivity loss).

**INV-013 note:** Phase 228 introduced `is_vhp_valid()` to chain.py. This contributed to
INV-013 digest drift (resolved in commit `29d5520c`, 2026-04-18).

## Gate Logic

```
if vhp_gated_invariant_change_enabled and category == "invariant_change":
    if not vhp_token_id:
        return 403 (missing token)
    result = chain.is_vhp_valid(vhp_token_id)   # fail-open: only blocks on confirmed False
    if result is False:
        return 403 (expired or invalid)
```

## Governance Design Note

Default is `False` (disabled) — fail-safe. The gate is opt-in to avoid blocking CI/CD
environments without a live VHP. When enabled, the gate is the first cryptographic link
between physical biometric presence (VHP) and protocol governance (invariant changes).
