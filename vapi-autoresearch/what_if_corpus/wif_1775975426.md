## W1 — Failure Mode: Prompt-Invariant Tests Are Static; Protocol Drift Makes Passing Tests
## Silently Wrong

### Root Cause Chain

Phase 201 introduced 8 prompt-invariant tests (T201-1..8) that verify LLM agent system
prompts contain required Phase 200 strings: `"Phase 200"`, `"0.728"`, `"TOURNAMENT BLOCKER"`,
`"7.009"`, `"5.367"`, `"dry_run=True"`, `"228"`. These tests function as a **regression
guard at commit time** — a future edit that removes a frozen invariant string will fail CI.

However, these tests have a structural gap that the mechanism itself cannot close:

**The invariant strings are correct at Phase 200 but will silently become stale at Phase N+K.**

When Phase 202 advances the protocol (e.g., separation ratio crosses 1.0, tremor_resting
probe adds new P0 conditions, or dry_run=False is activated after N≥100 adjudications):

1. The test file `test_phase201_agent_context.py` still checks for `"0.728"` — even if
   the real separation ratio is now `"1.043"`. The test **passes** (because the prompt
   still contains the old value as historical context), but the LLM now believes the current
   state is 0.728 when it is 1.043. The LLM's tournament guidance is wrong.

2. There is no runtime mechanism to detect when a live agent prompt is stale relative to
   the current protocol state in the database. The T201 tests run once at commit time;
   they do not run on a schedule or against live runtime state.

3. **Most critically**: when the protocol crosses a threshold that changes the LLM's
   recommended action (dry_run transitions to live mode, tournament becomes ACTIVE instead
   of BLOCKED), the agent will continue giving the old advice. The operator trusts the LLM
   answer without knowing the underlying prompt is from a prior phase.

### Implication

**LLM-agent semantic drift** — the prompt becomes progressively more wrong while all tests
continue to pass. This is not a code regression (the tests catch that); it is a **knowledge
regression** where the agent's ground truth diverges from protocol reality after each new
phase without being caught by any automated check.

**Severity escalation path:**
- Phase 202+: agent recommends `ALL_PAIRS_GATE_ENABLED=false` as the active path even after
  all_pairs_p0_ok is satisfied and the flag should be re-enabled.
- Post-dry_run activation: `SessionAdjudicatorAgent` continues emitting "advisory only — dry_run=True"
  on rulings that are now live enforcement.
- Post-separation-ratio-1.0: `CalibrationIntelligenceAgent` continues flagging TOURNAMENT BLOCKER
  after the tournament is authorized.

### Mitigation Available

**Yes** — two complementary mechanisms described in W2 below.

1. **Live prompt hash verification** at startup: compute `SHA-256(system_prompt)` for each
   LLM agent; compare to a committed hash stored in bridge DB (updated each phase by the
   Skill 14 PostCode sweep). If the stored hash diverges from the live hash, fire a
   `context_hash_mismatch` coherence event into `FleetSignalCoherenceAgent` — a new 4th
   INVERSION rule.

2. **On-chain prompt commitment** via `AgentContextRegistry.sol` (Phase 202): anchor
   `SHA-256(prompt_content + phase_number + timestamp)` as an immutable audit trail so
   tournament operators can cryptographically verify the LLM instructions that produced
   any ruling.

## W2 — Opportunity: AgentContextRegistry — World's First On-Chain LLM Instruction Audit Trail

### What It Is

An exclusive novel integration for VAPI: anchor the cryptographic hash of every LLM agent
system prompt in a dedicated on-chain registry. Combined with the existing
`FleetSignalCoherenceAgent` (Phase 193) and the Phase 201 PostCode sweep methodology, this
creates a complete **verifiable chain of custody for AI decision logic** — not just for
the data the AI evaluated, but for the instructions the AI was running when it evaluated it.

No other DePIN anti-cheat protocol (or any gaming anti-cheat system) has attempted this.

### Mechanism

**Phase 202 — AgentContextRegistry (new contract):**

```solidity
// AgentContextRegistry.sol
// Immutable audit trail for VAPI LLM agent system prompt versions.
// anchor(agentId, promptHash, phaseNumber) — onlyOperator
// getLatestHash(agentId) → (bytes32 hash, uint32 phase, uint256 timestamp)
```

- `promptHash = keccak256(abi.encode(sha256(promptContent), phaseNumber, chainId))`
- Anti-replay: UNIQUE(agentId, promptHash) — same prompt anchored twice is a no-op
- `agentId` is a bytes32 identifier: `keccak256("bridge_agent")`, `keccak256("session_adjudicator")`,
  `keccak256("calibration_intelligence_agent")` — stable across versions, human-readable

**Bridge DB — `agent_context_log` table (new):**
```sql
CREATE TABLE agent_context_log (
    id INTEGER PRIMARY KEY,
    agent_id TEXT NOT NULL,            -- "bridge_agent" etc.
    prompt_sha256 TEXT NOT NULL,       -- hex, 64 chars
    phase_number INTEGER NOT NULL,
    on_chain_tx TEXT,                  -- NULL until anchored
    anchored_at REAL,
    created_at REAL DEFAULT (unixepoch('now'))
);
CREATE UNIQUE INDEX ux_agent_context ON agent_context_log(agent_id, prompt_sha256);
```

**New store methods:**
- `upsert_agent_context_hash(agent_id, prompt_sha256, phase_number)` — idempotent INSERT OR IGNORE
- `get_agent_context_status(agent_id)` → latest hash, phase, on_chain_tx, anchored_at
- `get_all_agent_context_status()` → all 3 agents

**New bridge endpoint:**
- `GET /agent/context-integrity-status` — returns live hash vs committed hash for all 3 agents;
  `hash_match=True` when live == committed; `phase_current=True` when committed phase == current phase

**New Tool #150:**
- `verify_agent_context_integrity()` → calls `GET /agent/context-integrity-status`; if any
  `hash_match=False`, fires `CONTEXT_HASH_MISMATCH` coherence event into FSCA

**FleetSignalCoherenceAgent 4th INVERSION Rule (Phase 202 addition):**
- `CONTEXT_HASH_MISMATCH`: when live `SHA-256(system_prompt)` diverges from committed hash
  in `agent_context_log`. This is the bridge between the static Phase 201 test methodology
  and the runtime coherence monitoring of Phase 193.
- `delta = current_phase - committed_phase` when > 2 phases → SEMANTIC (prompt very stale)
- `delta = 1–2 phases` → STRUCTURAL (expected during active development, fix PostCode)
- `delta = 0` (same phase, hash changed) → SEMANTIC (unauthorized prompt edit — CRITICAL)

**Startup integration:**
At bridge startup (`main.py`), after all agents initialize, compute and store hashes:
```python
for agent_id, prompt in [
    ("bridge_agent", bridge_agent._SYSTEM_PROMPT),
    ("session_adjudicator", session_adjudicator._SYSTEM_PROMPT),
    ("calibration_intelligence_agent", calibration_intelligence_agent._CALIB_SYSTEM_PROMPT),
]:
    h = hashlib.sha256(prompt.encode()).hexdigest()
    store.upsert_agent_context_hash(agent_id, h, cfg.current_phase)
```

**On-chain anchoring (optional, controlled by `agent_context_on_chain_enabled=False` flag):**
When enabled, `POST /agent/anchor-context-hashes` triggers `chain.anchor_agent_contexts()`
which calls `AgentContextRegistry.anchor()` for each of the 3 agents. Tournament operators
can call `AgentContextRegistry.getLatestHash(agentId)` on IoTeX to verify which prompt
version produced any ruling with a given on-chain commitment hash — full chain of custody.

### Why This Is Exclusive to VAPI

This integration only makes sense in VAPI because:
1. VAPI already has PoAC commitment hashes + CeremonyAuditRegistry — the on-chain audit
   trail pattern is established. Adding prompt hashes extends the same chain of custody.
2. VAPI's LLM agents produce tournament rulings that have real-world consequences (player
   eligibility). Being able to prove to a player's legal representative "here is the exact
   AI instruction set that produced the ruling against you" is a compliance differentiator.
3. No competitor has both on-chain gaming anti-cheat AND LLM agent rulings — no other
   system has a reason to build this.
