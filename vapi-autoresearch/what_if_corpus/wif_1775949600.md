## W1 — Failure Mode: Silent Mahalanobis Inflation via USB Threshold Inheritance on BT Sessions

### Root Cause Chain

When `bt_transport_enabled` is set to `True` (Phase 120) and no BT-specific threshold track
exists in `l4_threshold_tracks`, the `l4_threshold_router.py` (Phase 126) falls back to global
thresholds: `anomaly=7.009`, `continuity=5.367`. These were calibrated on **N=74 USB sessions
at 1002 Hz**.

The L4 Mahalanobis distance is computed against a **12-feature covariance matrix** built from
USB data. Two features are critically sensitive to sampling rate:

1. **`tremor_peak_hz`** — FFT window of 1025 positions.
   - USB (1002 Hz): bin resolution = 0.977 Hz/bin → 4 bins cover the 8–12 Hz tremor band
   - BT (250 Hz): bin resolution = 0.244 Hz/bin → **16 bins** cover the same band
   - The tremor FFT accumulates 4× more bins across the physiological band. If the ring
     buffer is the same size (1024 frames), the BT window covers **4.096 seconds** vs USB
     **1.022 seconds**. Longer window = lower spectral leakage but different frequency
     resolution entirely — the peak bin index is numerically different between transports.
   - Result: `tremor_peak_hz` computed on BT data is **NOT comparable** to the USB-calibrated
     covariance. The feature mean and variance in the covariance matrix were built on
     USB-resolution peak indices (8–12 Hz → bins 8–12 at 0.977 Hz/bin). BT produces peak
     indices in the range 32–49 (same Hz, but 0.244 Hz/bin). The USB covariance matrix
     treats BT indices as extreme outliers → massive Mahalanobis inflation.

2. **`accel_magnitude_spectral_entropy`** — 1024-frame ring buffer, Shannon entropy of
   ||accel|| power spectrum from DC-removed samples.
   - USB ring of 1024 frames at 1002 Hz = **1.022 seconds** of data.
   - BT ring of 1024 frames at 250 Hz = **4.096 seconds** of data.
   - Longer windows exhibit lower spectral entropy (more low-frequency energy dominates).
   - USB calibration yielded mean 4.878–4.882 bits. BT will yield structurally lower entropy
     (~2.5–3.5 bits estimated due to longer integration period smoothing out high-frequency
     transients). USB covariance matrix has tight variance around ~4.88 bits; BT sessions
     land 1.5–2.0 σ below the USB mean → spurious L4 anomaly on every legitimate human BT session.

3. **`stick_autocorr_lag1` and `stick_autocorr_lag5`**
   - Lag is in samples. At USB (1002 Hz), lag1 = ~1ms, lag5 = ~5ms.
   - At BT (250 Hz), lag1 = 4ms, lag5 = 20ms.
   - Autocorrelation structure changes — lag5 in USB captures the same temporal interval as
     lag1.25 in BT. The feature values are physically different quantities at the two rates.

### Implication

If USB thresholds are inherited when BT is first enabled:
- **False positive rate**: structurally elevated — every legitimate BT session will fail L4
  due to `tremor_peak_hz` index mismatch and `accel_magnitude_spectral_entropy` depression.
- **False negative risk (more dangerous)**: a BT-aware bot that knows USB thresholds are
  active might inject signals calibrated to USB Mahalanobis norms. At 250 Hz, these
  injections produce USB-space distances below 7.009 even though the injection is blatant
  at 250 Hz — because the USB covariance matrix is measuring in a different sample-rate basis.
- The `l4_threshold_router.py` already logs a WARNING on global fallback (Phase 126 mitigation),
  but if `l4_battery_threshold_enabled=True` and a BT track was never created, the fallback
  silently applies USB values with no hard block.

### Mitigation Available

**Yes** — two-part:
1. `bt_transport_enabled=False` default prevents BT sessions from entering L4 evaluation.
   This is a HARD guard: never set `True` without a BT threshold track in `l4_threshold_tracks`.
2. When BT calibration sessions are collected (N≥50/player recommended), run
   `threshold_calibrator.py --battery bt_resting` (or `bt_gameplay`) to produce BT-specific
   thresholds. Insert via `POST /agent/l4-threshold-track` with `battery_type="bt_resting"`.
   The composite key `<profile_hash>:bt_resting:ble` routes all BT frames to this track.

**Proposed gate** (Phase 197+ candidate): Block `bt_transport_enabled=True` activation at
config load time when no active BT threshold track exists in `l4_threshold_tracks`.
`config.py` should raise `ValueError("BT_TRANSPORT_ENABLED requires bt_resting BT track")`.

## W2 — Opportunity: BT Minimum Viable Calibration Protocol (MVCP)

### Minimum Session Requirements

The BT calibration problem differs from USB in one critical dimension: **the FFT window covers
4× more time at 250 Hz**. This means:

- Per-session tremor stationarity is better at BT (longer averaging window).
- Fewer sessions are needed to establish stable `tremor_peak_hz` and `tremor_band_power`
  centroids per player.
- BUT: BT adds new variance sources — BLE radio jitter (~0.5–2ms latency variance per
  notification), potential frame dropping under interference, and OS BLE stack scheduling
  artifacts (Windows 11 BLE driver has ~1ms polling granularity).

**Recommended minimum:** N=30 BT sessions per player (vs N=74 for USB), structured as:
- 10× `bt_resting_grip` (controller at rest, BLE connected) — establishes `accel_magnitude_spectral_entropy` baseline
- 10× `bt_touchpad_corners` — validates touch feature parity with USB corpus
- 10× `bt_gameplay` (active NCAA CFB 26 session) — establishes `tremor_peak_hz` + `stick_autocorr` under gaming load

**Minimum viable for threshold derivation (not tournament gate):** N=10 per player (3 probe
types, balanced) using `threshold_calibrator.py --battery bt_resting`.

### Which Probe Types Are Valid at 250 Hz

| Probe type | Valid at 250 Hz? | Notes |
|------------|-----------------|-------|
| `bt_resting_grip` | YES | Lower polling doesn't matter for resting tremor |
| `bt_touchpad_corners` | YES | Touchpad at 250 Hz: spatial resolution unchanged (touch controller independent of BLE IMU rate) |
| `bt_gameplay` | YES | Primary use case; R2 trigger timing affected by 4ms BT frame |
| `bt_micro_tremor` | CAUTION | tremor_peak_hz bins differ; must compute BT-native FFT, not USB-ported |
| `touchpad_freeform`/`touchpad_swipes` | YES (equivalent) | Touch events unaffected by IMU rate |

**Invalid to reuse from USB corpus:** `tremor_peak_hz`, `stick_autocorr_lag1/5`, `accel_magnitude_spectral_entropy` reference values. Must be re-derived from BT sessions.

### Recommended Threshold Derivation

1. Collect N≥30 BT sessions (10 per probe type, 3 players balanced).
2. Run `threshold_calibrator.py --battery bt_resting --transport ble`:
   - The calibrator must re-compute covariance on BT-only sessions.
   - It must NOT mix USB and BT sessions in the same covariance matrix.
   - Output: `bt_anomaly` and `bt_continuity` thresholds (expected: lower than USB, ~5.5–6.5 anomaly
     and ~4.0–5.0 continuity, because BT's longer FFT window reduces high-frequency variance).
3. Insert track: `battery_type="bt_resting"`, `transport_type=TRANSPORT_TYPE_BLE`.
4. Verify `l4_threshold_router.py` composite key routing: `<profile_hash>:bt_resting:ble` → BT track.
5. Only then set `bt_transport_enabled=True` with `l4_battery_threshold_enabled=True`.

### Phase Candidate

**Phase 197+ BT Calibration Gate:**
- Add `bt_calibration_sessions_needed` to `GET /agent/l4-calibration-status` response.
- Block `bt_transport_enabled=True` via config validation when no BT track exists.
- New `bt_calibration_log` table to track BT session collection progress.
- `score_phase_bt_readiness()` gate: requires N≥30 BT sessions, separate covariance,
  and BT threshold track active before tournament eligibility check runs for BT-transport devices.

**Effort:** ~1 day implementation + hardware dependency (requires BLE pairing + N=30 sessions
across 3 players). Pre-blocked until BLE sessions captured. Estimated bridge +8, SDK +4 tests.

## BT Feature Translation Reference

| Feature (index) | USB behavior | BT behavior | Comparable? |
|-----------------|-------------|-------------|-------------|
| `tremor_peak_hz` (8) | FFT bin at 1002 Hz; bins 8–12 = 8–12 Hz | FFT bin at 250 Hz; bins 32–49 = 8–12 Hz | NO — raw index differs |
| `tremor_band_power` (9) | 4 bins summed across 8–12 Hz | 16 bins summed | NO — magnitude different |
| `accel_magnitude_spectral_entropy` (9) | 1.022s window, mean ~4.88 bits | 4.096s window, estimated ~2.5–3.5 bits | NO — rate-dependent |
| `stick_autocorr_lag1` (5) | lag=1ms | lag=4ms | NO — lag semantics differ |
| `stick_autocorr_lag5` (6) | lag=5ms | lag=20ms | NO — lag semantics differ |
| `micro_tremor_accel_variance` (3) | 1002 Hz variance | 250 Hz variance (lower; smoothed) | NO — scale differs |
| `grip_asymmetry` (4) | ratio of accel axes | same computation | APPROXIMATE |
| `touch_position_variance` (10) | touchpad HW independent | same | YES |
| `press_timing_jitter_variance` (11) | IBI in samples (1/1002s) | IBI in samples (1/250s) | CAUTION — unit differs |
| `touchpad_spatial_entropy` (12) | 8×8 grid, same | same | YES |

**Safe to reuse from USB corpus:** `touch_position_variance`, `touchpad_spatial_entropy`, `grip_asymmetry`.
**Must recalibrate for BT:** all spectral/temporal features (6 of 13 active features).
