## W1 — Failure Mode: Declining Separation Ratio Velocity Triggers Premature SeparationRatioRegistry.sol Commitment

### Root Cause Chain

The touchpad_corners corpus demonstrated a critical pattern: ratio peaked at 0.998 (N=29)
then declined to 0.728 (N=35) as P3 intra-player variance (intra_mean=0.802) dominated the
interperson Mahalanobis distance computation. This was not an outlier event — it was a
structural property of P3 non-stationarity manifesting as the corpus grew.

When tremor_resting sessions approach ratio≈1.0, two irreversible actions will be triggered:

1. **IoSwarmConsensusAggregator MINT_QUORUM=0.80** will authorize VHP mint based on
   quorum-confirmed separation_ok=True
2. **SeparationRatioRegistry.sol** will accept the on-chain SHA-256 commitment hash anchored
   to the ratio value at the moment of commitment

If the ratio subsequently declines (as it did in touchpad_corners: 0.998→0.728), the
on-chain commitment is cryptographically correct at time of writing but legally and
operationally indefensible for tournament authorization. The Phase 153/197 per-pair gate
(`all_pairs_above_1`) mitigates the per-pair floor but does **not** detect a declining
velocity trend before the IoSwarm quorum fires.

The protocol currently has **no velocity gate** between ratio≈1.0 and the irreversible
commitment chain.

### Implication

- IoSwarm MINT_QUORUM=0.80 quorum-authorized VHP mints could occur at a separation ratio
  that subsequently declines below 1.0 due to P3 non-stationarity
- SeparationRatioRegistry.sol commitments are immutable (anti-replay UNIQUE commitHash)
- A tournament operator auditing 6 months post-issuance sees `ratio=1.043` on-chain;
  actual live ratio is 0.79. The `isFullyEligible()` composability guarantee is falsified
- The `dry_run=True` enforcement gate cannot prevent this: VHP mints fire before dry_run=False
  is authorized; the commitment precedes enforcement activation by design

### Severity

**P0 — TOURNAMENT BLOCKER ADJACENT**. Not a current blocker (ratio=0.728 is far from 1.0),
but becomes critical when ratio approaches 0.90+. Must be gated before approach.

### Mitigation Available

**YES** — TremorRestingConvergenceOracle (W2 below).

Compute `convergence_velocity = (ratio_current - ratio_prev) / N_delta` after each
tremor_resting session. Gate conditions for SeparationRatioRegistry.sol commitment:
1. `convergence_velocity >= 0` (ratio is not declining)
2. `ratio > 1.0`
3. `all_pairs_above_1=True`
4. IoSwarmConsensusAggregator `BLOCK_QUORUM=0.67` confirms convergence_stable

If `velocity < 0` for 2 consecutive sessions: fire `RATIO_VELOCITY_NEGATIVE` coherence
event into FleetSignalCoherenceAgent as a new ORPHAN rule (5th rule addition to Phase 193's
5-rule ORPHAN set). Block VHP quorum mint until velocity recovers positive.

`dry_run=True` enforcement gate extension: never authorize `dry_run=False` transition while
`convergence_declining=True` — this prevents the scenario where enforcement goes live on a
ratio that is actively declining toward <1.0.

## W2 — Opportunity: TremorRestingConvergenceOracle + Class K_t Tremor-Frequency Spoofer Definition

### W2a — TremorRestingConvergenceOracle

**What It Is:** A per-session velocity monitor for the tremor_resting separation ratio that
gates the irreversible SeparationRatioRegistry.sol commitment chain. Closes the touchpad_corners
failure mode for the tremor_resting protocol.

**Mechanism:**

After each tremor_resting session is added to the interperson separation corpus:
1. Recompute separation ratio with proper LOO using `tremor_peak_hz` as primary discriminator
2. Fit convergence velocity from last 3 snapshots: `velocity = Δratio / ΔN_sessions`
3. If `velocity >= 0` for 2 consecutive sessions: fire `tremor_convergence_stable` bus event;
   IoSwarmConsensusAggregator (BLOCK_QUORUM=0.67) validates and stores consensus
4. If `velocity < 0` for 2 consecutive sessions: fire `convergence_declining` ORPHAN rule
   into FleetSignalCoherenceAgent; block VHP MINT_QUORUM=0.80 from firing
5. Persist in `tremor_convergence_log` table (session_type, ratio, velocity, convergence_stable)
6. `GET /agent/tremor-convergence-status` returns convergence_stable, velocity, sessions_needed_estimate

**Sessions-needed estimate:** simple linear extrapolation from current velocity — gives
operator a concrete target number before committing to more capture sessions.

**New store methods:**
- `insert_tremor_convergence_log(session_type, ratio, velocity, stable)` → int
- `get_tremor_convergence_status()` → latest convergence state

**Phase Candidate:** Phase 202 — +8 bridge tests, +4 SDK tests, no Hardhat needed.
`tremor_convergence_enabled=False` default (infrastructure-first pattern).

**Exclusive to VAPI because:**
- Requires VAPI per-session Mahalanobis separation infrastructure
- Requires IoSwarmConsensusAggregator BLOCK_QUORUM=0.67 quorum validation
- Requires SeparationRatioRegistry.sol on-chain commitment chain (Phase 153)
- No other DePIN protocol has adaptive separation velocity monitoring

### W2b — Class K_t: Mechanical Tremor-Frequency Spoofer Threat Model

**What It Is:** The first concrete definition of the Class K hardware biometric spoofer
threat surface, scoped specifically to tremor_peak_hz as the primary separation discriminator.

**Threat Model:**

`Class K_t` (tremor-frequency spoofer): a small vibration motor (e.g., ERM coin motor,
3.7V, 9000 RPM = 150 Hz) attached to the DualShock Edge grip, controlled by a microcontroller
to reproduce a target player's `tremor_peak_hz` signature (e.g., P1 ≈ 9.37 Hz).

**Attack surface:**
- Motor runs at constant, programmable frequency independent of game events
- Physiological tremor is coupled to central nervous system → correlates with stress,
  game-state changes, decision-making load
- Motor-induced tremor does **not** correlate with game events — it's mechanical noise

**Detection approach:** `tremor_gsr_correlation` — physiological tremor correlates with
GSR SCR (skin conductance response) peaks during high-stress game events. Motor-induced
tremor shows `gsr_game_event_correlation ≈ 0.0` while physiological tremor shows
`gsr_game_event_correlation > 0.3`.

**New advisory inference code:** `0x34 TREMOR_DECOUPLED`
- Trigger: `tremor_peak_hz` in target band (7–12 Hz, player-specific) AND
  `gsr_game_event_correlation < 0.1` (no physiological coupling)
- Type: ADVISORY ONLY — never hard gate until N≥30 GSR calibration per player
- `MockTremorMotor(freq_hz, seed=42)` for code-first development (same precedent as MockGSRGrip)

**Phase Candidate:** Phase 203 — Class K_t threat model + 0x34 TREMOR_DECOUPLED inference
code + MockTremorMotor. +8 bridge tests, +4 SDK tests. `class_k_enabled=False` default.

**Exclusive to VAPI because:**
- Requires simultaneous tremor_peak_hz (L4 feature) + GSR SCR correlation (L7 feature) +
  game event timeline (L5 temporal rhythm) — three signals only VAPI's stack has together
- No other DePIN anti-cheat system has defined a hardware-level biometric spoofing attack
  with a cryptographic countermeasure grounded in IoSwarm quorum consensus validation
- The MockTremorMotor precedent (code-before-hardware) is VAPI-native pattern
